各维度详解
5个维度逐一拆解：计算公式、业务故事、可行性评估、数据来源
SAP 全自动 D1 · 回款及时率
20%→25%
权重 25% · 连续值评分
业务含义（业务人员语言）
这个指标回答一个最基本的问题：我们把货发出去、把模具交付了，钱能不能按时收回来？超过30天没收到的款项代表催款风险，超过60天则代表坏账风险。
数据来源与可行性
来自 SAP 应收账款账龄报表，每月月结后系统自动推送。现有系统完全支持，零开发成本，仅需配置定时导出接口。
业务故事举例
客户 A 总账期 100 万，其中 20 万超过 30 天未还、5 万超过 60 天未还。超30天比例 = 20%，超60天比例 = 5%。原始分 = 100 − 20×10 − 5×20 = 0 分。意味着这个客户的回款风险已到临界点，D1 = 0。
计算公式（含V4.0升级：权重25%）
-- D1 回款及时率（V4.0 权重提升至 25%） -- 数据来源：SAP 应收账款账龄报表，每月自动推送 超30天比例 = 逾期30-60天应收余额 / 该客户总应收余额 × 100 （%） 超60天比例 = 逾期60天以上应收余额 / 该客户总应收余额 × 100 （%） -- 互锁约束：超60天不得大于超30天 ASSERT: 超60天比例 ≤ 超30天比例 D1原始分 = MAX(0, 100 - 超30天比例 × 10 - 超60天比例 × 20 ) D1得分 = D1原始分 / 20 → 取值范围 [0, 5]，连续值，非阶梯 D1贡献 = D1得分 × 4 × 战略权重 → 满分贡献 20 分 -- 边界处理：总应收为零时（新客户/无业务期），D1得分 = 5（满分，视为无风险） IF 总应收余额 = 0: D1得分 = 5.0
✓
权重提升理由：D1 是唯一纯财务安全指标，且完全由 SAP 客观数据驱动。在人工指标权重压缩后，将其从 20% 提升至 25% 符合"提高客观指标话语权"的优化目标，同时补偿了 D3 替换带来的结构变化。
SAP 全自动 D2 · 项目利润率
20% 3档→25% 5档
权重 25% · 1–5分五档
业务含义（业务人员语言）
这个指标回答：这个客户的项目赚钱吗？赚多少？不同业务类型（模塑一体 vs 商品模）利润基准不同，系统自动根据客户类型切换评分阈值。
V4.0 关键升级：增加中间档
原来只有1/3/5三档，新增 2分档和 4分档，使利润率微小变化也能反映在得分上，避免"利润从25%降到20.1%得分完全不变"的问题。
业务故事举例
模塑一体客户 B，过去12个月综合毛利率 22%，落在 20%–25% 区间，D2 = 4分。上季度因原材料涨价，毛利率降至 18.5%，落入 15%–20% 区间，D2 降为 3分，健康度自动下调，触发成本工程师复盘。
评分规则（5档）
-- D2 项目利润率（V4.0 新增 2/4 档，提升敏感度） -- 数据来源：SAP 项目结算表，滚动12个月加权平均毛利率 综合毛利率 = Σ(各项目毛利额) / Σ(各项目收入) × 100 （%，滚动12月） IF BIZ_TYPE = '模塑一体': 毛利率 > 25% → D2 = 5 （超额利润，优质） 22% < 毛利率 ≤ 25% → D2 = 4 （良好）★新增档 18% < 毛利率 ≤ 22% → D2 = 3 （正常） 15% < 毛利率 ≤ 18% → D2 = 2 （偏低）★新增档 毛利率 ≤ 15% → D2 = 1 （告警） IF BIZ_TYPE = '商品模': 毛利率 > 35% → D2 = 5 30% < 毛利率 ≤ 35% → D2 = 4 ★新增档 24% < 毛利率 ≤ 30% → D2 = 3 20% < 毛利率 ≤ 24% → D2 = 2 ★新增档 毛利率 ≤ 20% → D2 = 1 -- 异常处理：项目数不足3个时，加注"数据不足"标记，得分仍计算但标黄提示 IF 滚动12月项目数 < 3: 标记 D2_DATA_WARNING = TRUE
SAP 全自动 ★全新 D3 · 客户活跃度指数
SOB人工→活跃度自动
权重 20% · 连续值评分
★
为什么替换 D3？原 SOB（客户口袋占比）需要人工维护"客户市场总采购额"，这个数字难以获取、更新滞后，客户经理普遍抗拒填写，已成为系统落地最大阻力。新 D3「客户活跃度」完全基于 SAP 已有数据自动计算，零人工干预。
业务含义（业务人员语言）
这个指标回答：客户近期是不是一直在跟我们做生意？综合看三件事：最近有没有在下单、下单频率稳不稳定、下单金额有没有在增长。三者加权合成一个"活跃度分"。
数据来源与可行性
全部来自 SAP 销售订单和发货记录，系统现有数据，无需任何人工录入。开发成本：仅需封装3个 SAP 查询逻辑，约1–2个工作日。
业务故事举例
客户 C，近6个月每月都有发货（月均活跃 = 满分），PO 连续12个月（连续性满分），但金额同比下降8%（增长率得2分）。活跃度综合分 = 4.2，接近优质水平，D3 = 4.2。
计算公式（三子项加权）
-- D3 客户活跃度指数（V4.0 全新，替代人工 SOB） -- 由三个子指标加权合成，全部来自 SAP，无人工干预 -- 子项 A：近6个月月均活跃率（权重 40%） -- 含义：近6个月里，每个月都有发货或有 PO 的比例 活跃月数 = 近6个月中，有发货记录或有PO的月份数 （最大=6） A得分 = (活跃月数 / 6) × 5 → 取值 [0, 5] -- 子项 B：PO 连续稳定性（权重 40%） -- 含义：过去12个月，有几个月有PO（连续性越高越好） 连续PO月数 = 过去12个月中，有采购订单的月份数 （最大=12） B得分 = (连续PO月数 / 12) × 5 → 取值 [0, 5] -- 子项 C：采购金额同比变化（权重 20%） -- 含义：今年比去年同期多还是少 金额同比变化% = (近12月金额 - 前12月金额) / 前12月金额 × 100 C得分 = CASE WHEN 金额同比变化% > 15 : 5 （明显增长） WHEN 金额同比变化% > 0 : 4 （小幅增长） WHEN 金额同比变化% > -10 : 3 （基本持平） WHEN 金额同比变化% > -20 : 2 （小幅下降） ELSE : 1 （明显下降） END -- 合成 D3 得分 D3得分 = A得分 × 0.4 + B得分 × 0.4 + C得分 × 0.2 → 取值 [0, 5] -- 边界处理：新客户（前12月无数据），C得分默认3分（中性），标注"历史数据不足" IF 前12月金额 = 0: C得分 = 3, 标记 D3_NEW_CUSTOMER = TRUE
✓
业务合理性：活跃度指数比"口袋份额"更直接反映当下的合作健康程度。一个 SOB 很高但近期订单断档的客户，活跃度指数会立刻降低并触发预警；而 SOB 指标因数据滞后，可能数月后才能反映问题。
SAP辅助+人工 D4.1 · 合作稳定性
20% 3档→15% 五档
权重 15% · 1–5分五档
⚠
V4.0 关键改进：SAP 自动生成"建议分值"，客户经理只需确认或微调（±1档），无需从零填写。若实际填写与系统建议差距超过1档，自动触发销售经理复核。人工操作从"主动填写"变为"被动确认"，阻力大幅降低。
业务含义（业务人员语言）
这个指标看：我们和这个客户的合作是在加深还是在收缩？新项目在增加、还是已经多年没有新动作了？
为什么保留人工？
SAP 看到的是历史数据，但客户经理知道未来的信号，例如"客户已口头确认明年新项目"或"总监已离职、关系需要重建"。这些信息只有人知道，因此保留人工微调权。
业务故事举例
系统根据 SAP 数据建议 D4.1 = 3分（稳定存量）。但客户经理知道上月刚签了两个新模合同（尚未在 SAP 入账），将其调整为 4分，并备注"新模合同参考单号 XXX"。调整幅度在允许范围（±1档），无需复核。
SAP自动建议值 + 人工确认规则
-- D4.1 合作稳定性（V4.0：SAP建议 + 人工微调，权重降至15%） -- Step 1: SAP 自动计算建议分值 近6月新项目数 = COUNT(近6个月内新启动的项目号，来自SAP) PO连续月数 = 过去12个月有PO的月份数（来自SAP） 近6月有无发货 = SAP 发货记录是否为空 -- 建议分值规则（系统自动生成，展示给客户经理） 建议分 = CASE WHEN 近6月新项目数 >= 2 : 5 （强势增长） WHEN 近6月新项目数 = 1 : 4 （稳步推进） WHEN PO连续月数 >= 10 AND 近6月新项目数 = 0 : 3 （稳定存量） WHEN PO连续月数 BETWEEN 6 AND 9 : 2 （轻度萎缩） WHEN 近6月有无发货 = FALSE : 1 （业务停滞） END -- Step 2: 客户经理确认/微调（每季度一次，仅需操作1次） 实际D4.1 = 客户经理填写值 （允许在建议分 ±1档内自行调整） -- Step 3: 差异检查（超过1档差异触发复核） IF ABS(实际D4.1 - 建议分) > 1: → D4_1_REASON 字段变为必填（填写调整原因，如"新合同尚未入账"） → D4_1_APPROVED_BY 字段变为必填（销售经理签字确认）
CRM 硬锁 D4.2 · 质量表现
20% 3档→15% 五档+硬锁
权重 15% · 1–5分五档
业务含义（业务人员语言）
这个指标问：客户投诉我们多不多？严不严重？没有客诉最好；有客诉但不严重、属于偶发，可以接受；反复投诉或有严重质量事故，就要拉响警报了。
V4.0 关键升级：准自动化
系统每季度从 CRM 客诉工单模块自动统计数量和等级，生成预填分值。客诉 ≥ 3 单强制锁定为 1分，无需人工干预，消除"人情分"。
业务故事举例
客户 D 本季度有 2 张客诉工单，均为轻微缺陷已关闭。系统预填 D4.2 = 3分，客户经理看了一眼确认，全流程不到30秒。若有1张为重大质量事故，系统直接锁定 1分，客户经理无法修改。
系统自动评分 + 客诉硬锁规则（强制保留）
-- D4.2 质量表现（V4.0：五档细化 + 硬锁加强，权重降至15%） -- 数据来源：CRM 客诉工单模块，按客户编号统计近6个月 -- Step 1: 系统自动统计（每季度触发） 客诉总数 = CRM近6个月该客户名下客诉工单数 重大客诉数 = 其中严重级别标记为"重大"或"停线"的工单数 重复缺陷数 = 同一缺陷类型在近6个月内出现 ≥ 2 次的工单数 -- Step 2: 硬锁规则（不可被人工覆盖，强制锁定） IF 客诉总数 >= 3 OR 重大客诉数 >= 1 OR 重复缺陷数 >= 1: 强制锁定 D4.2 = 1 ← 任何人无法修改，包括销售经理 -- Step 3: 非锁定情况下的五档评分（系统预填，人工确认） 预填分值 = CASE WHEN 客诉总数 = 0 : 5 （零客诉，优质） WHEN 客诉总数 = 1 AND 均为轻微客诉 : 4 （偶发轻微） WHEN 客诉总数 = 2 AND 均为轻微客诉 : 3 （有待改善） WHEN 客诉总数 = 2 AND 有中等级别 : 2 （较差） -- 注：硬锁条件未触发时不会出现1分，1分由硬锁专用 END -- Step 4: 客户经理点击"确认"即完成（无需文字输入，仅需确认预填值） -- 若不同意预填值，可在非硬锁范围内调整，需填写原因（必填字段）
🔒
客诉硬锁机制（必须保留，不可妥协）：这是整个 D4.2 客观性的最后防线。若取消硬锁，客户经理可以无限给问题客户打高分，导致健康度系统完全失去公信力。建议在系统层面通过数据库约束（而非前端校验）实现，防止任何技术绕过。
